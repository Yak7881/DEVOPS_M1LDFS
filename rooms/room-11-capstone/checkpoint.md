## Mini-quiz (5 questions — révision de toutes les rooms)

**1.** **Git** : Quelle est la différence entre `git commit` et `git push` ?

**2.** **Branches** : Pourquoi crée-t-on une branche pour une nouvelle fonctionnalité au lieu de coder directement sur `main` ?

**3.** **Pull Request** : À quoi sert une PR dans le workflow d'équipe ?

**4.** **Pipeline** : Que se passe-t-il quand un pipeline CI devient rouge après un push ?

**5.** **Release** : Que signifie le format de version v1.2.3 (semantic versioning) ?

<details>
<summary>Voir les réponses</summary>

1. `git commit` sauvegarde tes modifications en local. `git push` envoie tes commits vers le dépôt distant (ex : GitHub).
2. Pour isoler ton travail sur main, éviter les conflits et permettre la revue de code via une PR avant fusion.
3. Une PR permet de proposer la fusion de ta branche dans main et sert d'endroit pour la revue de code par l'équipe.
4. Le pipeline devient rouge si un test échoue ou une étape échoue. Il faut corriger le code avant de merger ou déployer.
5. v1.2.3 = major.minor.patch : major = changements incompatibles, minor = nouvelles fonctionnalités, patch = corrections de bugs.

</details>
